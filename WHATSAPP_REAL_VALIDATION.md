# WhatsApp Real Validation Runbook

## Purpose

This document defines the operational validation steps required to prove the POC works with a real Meta WhatsApp Cloud API integration.

The goal is not to add new features. The goal is to confirm that the existing implementation works end to end with a real WhatsApp sender, a public webhook URL, and a live Meta configuration.

## Validation Scope

This runbook validates these POC requirements:

- Meta webhook verification works
- inbound WhatsApp text messages reach the application
- Gemini generates a reply
- the application sends a text reply back through WhatsApp Cloud API
- short conversation context is preserved
- basic logs and metrics reflect the live flow

This runbook does not validate advanced production concerns such as durable queues, multi-tenant routing, dashboards, or media message support.

## Prerequisites

Before starting, make sure you have:

- a reachable public HTTPS URL for this application
- access to a Meta app with WhatsApp Cloud API enabled
- a WhatsApp sender number or Meta test number configured in the app
- a recipient phone number allowed by the WhatsApp test setup, when applicable
- Railway or another deploy target with environment variables configured

## Required Environment Variables

The deployed application must have these values configured:

- `WHATSAPP_VERIFY_TOKEN`
- `WHATSAPP_APP_SECRET`
- `WHATSAPP_ACCESS_TOKEN`
- `WHATSAPP_PHONE_NUMBER_ID`
- `GEMINI_API_KEY`

Recommended for validation visibility:

- `REDIS_URL`
- `DATABASE_URL`
- `WEBHOOK_QUEUE_WORKERS`
- `WEBHOOK_QUEUE_BUFFER_SIZE`
- `WEBHOOK_QUEUE_MAX_RETRIES`
- `WEBHOOK_QUEUE_RETRY_DELAY`

## Deployment Readiness Checks

Before touching Meta settings, validate the deployed service directly.

### 1. Health endpoint

Call:

```text
GET /healthz
```

Expected result:

- HTTP `200`
- `whatsapp_sender_configured = true`
- `whatsapp_webhook_configured = true`
- `gemini_configured = true`

### 2. Metrics endpoint

Call:

```text
GET /metrics
```

Expected result:

- HTTP `200`
- JSON response with counters for webhook, queue, retry, and simulation activity

### 3. Local simulation still works

Call:

```text
POST /simulate
```

Expected result:

- HTTP `200`
- non-empty `reply_message`

This confirms the application logic works before involving Meta.

## Meta Webhook Configuration

When Meta access is available again, configure the webhook with:

- callback URL: `https://<your-domain>/webhook`
- verify token: the same value used in `WHATSAPP_VERIFY_TOKEN`

Subscribe the app to message-related webhook events required by the WhatsApp Cloud API flow.

Operational success criteria for verification:

- Meta calls `GET /webhook`
- the application returns the `hub.challenge`
- Meta marks the webhook as verified

## End-to-End Validation Steps

### Step 1. Send a real inbound message

Send a WhatsApp text message from the allowed recipient phone to the configured sender number.

Example user message:

```text
Hello, I need help renewing my registration.
```

Expected result:

- Meta delivers the webhook to `POST /webhook`
- the application returns HTTP `200`
- the message is enqueued for async processing
- a worker processes the message and sends a reply

### Step 2. Confirm logs

Inspect application logs.

Expected log events:

- webhook message accepted and enqueued
- webhook message processed by a worker
- no permanent failure for the message

If the sender is not fully configured, you may see a structured log indicating that outbound WhatsApp sending was skipped. That is not a valid end-to-end pass for real validation.

### Step 3. Confirm metrics

Call:

```text
GET /metrics
```

Expected counter movement:

- `webhook_requests` increases
- `webhook_messages` increases
- `webhook_enqueued` increases
- `webhook_processed` increases
- `webhook_failures` does not increase for the happy path

### Step 4. Confirm reply delivery on WhatsApp

Check the recipient phone.

Expected result:

- the user receives a text reply generated by the application
- the reply is coherent with the user message

### Step 5. Confirm short context retention

Send two related messages in sequence.

Example:

```text
Message 1: I need help renewing my registration.
Message 2: Which documents do I need?
```

Expected result:

- the second answer reflects the prior message context
- the conversation behaves as a short multi-turn exchange

## Idempotency Validation

To confirm duplicate delivery protection:

- trigger or replay the same webhook payload with the same Meta `message_id`
- or observe duplicate delivery behavior if Meta retries the same event

Expected result:

- the duplicate message is not processed twice
- `webhook_duplicates` increases
- the user does not receive two replies for the same inbound message

## Retry Validation

To confirm async retry behavior:

- temporarily induce a recoverable failure in reply generation or outbound sending
- then restore the dependency before retries are exhausted

Expected result:

- `webhook_retries` increases
- the job is retried by the background worker
- once the dependency is restored, the message is processed successfully
- persisted chat history is not duplicated by the retry path

## Acceptance Criteria

The POC is operationally validated when all of the following are true:

- Meta verifies `GET /webhook` successfully
- a real WhatsApp text message reaches `POST /webhook`
- the application sends a real text reply back through WhatsApp Cloud API
- the second user message benefits from short conversation context
- logs show enqueue and processing activity
- metrics reflect request, enqueue, processing, and duplicate handling behavior

## Failure Checklist

If validation fails, check these first:

- `WHATSAPP_VERIFY_TOKEN` does not match the Meta webhook configuration
- `WHATSAPP_APP_SECRET` is wrong, causing signature validation to fail
- `WHATSAPP_ACCESS_TOKEN` is expired or invalid
- `WHATSAPP_PHONE_NUMBER_ID` is incorrect
- the public webhook URL is not reachable over HTTPS
- the recipient phone is not allowed in the WhatsApp test setup
- `GEMINI_API_KEY` is missing or invalid
- queue settings are too small and enqueue is failing under load

## Evidence to Capture

For a successful POC sign-off, capture:

- screenshot or export of successful Meta webhook verification
- example inbound user message
- example outbound assistant reply on WhatsApp
- `/healthz` response
- `/metrics` snapshot after a successful test
- log lines showing enqueue and worker processing for the tested `message_id`
